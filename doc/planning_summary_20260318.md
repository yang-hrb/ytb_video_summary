**总评**：核心难点在 2.2 和 2.3——修改正在运行的核心 pipeline 逻辑是最危险的操作。每个 `process_*` 函数有不同的异常路径，`current_stage` 变量必须在所有代码路径上正确维护。resume 逻辑的边界条件多（音频文件已删除、SRT 存在但内容截断、数据库状态与磁盘文件不一致等）。

**架构师建议**：先为 `process_video()` 做 Step 2 改造并充分测试，再推广到其他两个函数。不要同时改三个。

### Step 5：定时巡检自动化

| 子任务 | 改动量 | 难度 | 风险 | 说明 |
|--------|--------|------|------|------|
| channel_watcher.py 核心模块 | ~200 行 | ★★★ | **中** | 频道视频列表抓取依赖 yt-dlp 的非官方行为，可能被 YouTube 限制 |
| watchlist 管理（文件或 DB） | ~50 行 | ★★ | 低 | CRUD 操作 |
| 增量扫描逻辑 | ~80 行 | ★★★ | **中** | 需准确判断"新视频"，upload_date 可能不准确或缺失 |
| 巡检状态持久化 | ~40 行 | ★★ | 低 | DB 表扩展 |
| crontab / schedule 集成 | ~30 行 | ★★ | 低 | 基础设施 |
| 异常处理 & 限流保护 | ~50 行 | ★★★ | **中** | YouTube 反爬策略可能导致巡检失败 |

**总评**：技术实现本身不复杂，但 **运维可靠性是核心挑战**。YouTube 的反爬机制可能导致频繁巡检被封 IP 或 cookie 失效。`yt-dlp` 获取频道视频列表是非官方 API 调用，稳定性无法保证。

**架构师建议**：
1. 默认巡检频率不宜高于每天 1 次，且对每个频道增加随机延迟
2. 增加 `--dry-run` 模式：只检查新视频但不处理，方便调试
3. 巡检失败时指数退避，连续 3 次失败暂停该频道
4. 考虑先支持 10 个频道的小规模验证，确认稳定后再扩展

## 四、架构师视角：整体设计评估

### 4.1 设计 Scope 评估

```
                    当前状态          →    5步完成后
                    ──────────            ──────────
核心 Pipeline:      ✅ 功能完整            ✅ 更健壮（阶段追踪 + 断点续传）
输入源覆盖:         ✅ 5种输入模式          ✅ + 自动巡检 YouTube 频道
转录引擎:           ⚠️ 仅 openai-whisper   ✅ 双后端（mlx + openai）
数据持久化:         ⚠️ 基础 SQLite          ✅ file_storage + 产物路径追踪
输出组织:           ⚠️ 扁平时间戳命名       ✅ 按 up主/月份 层级组织
可观测性:           ❌ 几乎没有             ✅ Daily Summary + CLI 诊断 + 精确错误定位
自动化:             ❌ 纯手动              ✅ crontab 定时巡检
测试:              ❌ 覆盖极低             ⚠️ 方案中提到但未列为独立 Step
```

### 4.2 架构优缺点

**优点**：
- 5 步方案循序渐进，依赖关系清晰（1→2→3→4→5）
- 每步都有向后兼容设计（DB 迁移、配置 fallback、路径 fallback）
- Step 3 的适配层设计（策略模式）是正确的抽象层次
- Step 2 的状态机设计完整，覆盖了所有 pipeline 阶段

**风险与不足**：

| 风险 | 严重度 | 说明 |
|------|--------|------|
| **main.py 持续膨胀** | 🔴 高 | 5 步方案都在往 main.py 塞逻辑（阶段追踪、文件注册、up主信息、daily summary CLI），但 Step 4 的"Pipeline 抽象重构"只是提到了但没有作为正式步骤。完成后 main.py 可能膨胀到 1500+ 行。 |
| **测试债务累积** | 🔴 高 | 每个 Step 都提到"验证计划"，但没有独立的测试建设步骤。5 步改完后核心 pipeline 复杂度翻倍，但测试覆盖率可能仍然很低。 |
| **Step 4 scope 过大** | 🟡 中 | 一个 Step 包含 9 个子任务、3 个层面的变更，实际是 2-3 个独立 Step 的工作量。 |
| **Step 5 外部依赖风险** | 🟡 中 | 依赖 YouTube 非官方 API 行为，长期稳定性存疑。 |
| **DB 设计碎片化** | 🟡 中 | runs 表通过 ALTER TABLE 逐步加列（Step 2 加 transcript_path 等，Step 4 又加 file_storage 表），没有一次性的 schema 设计。后续可能需要再次重构 DB。 |

### 4.3 架构师建议：调整后的实施路线

```
Phase 1 (基础清理)    Phase 2 (核心强化)    Phase 3 (体验提升)    Phase 4 (自动化)
──────────────────   ──────────────────   ──────────────────   ──────────────
Step 1: 清理+Bug修复  Step 2: 状态追踪      Step 4a: file_storage  Step 5: 定时巡检
                     Step 3: mlx-whisper    + 文件名重构
                     ★ 新增: Pipeline       Step 4b: GitHub重构
                       抽象重构              + Daily Summary
                     ★ 新增: 核心测试建设    + up主元信息

预计: 1-2 天          预计: 5-7 天          预计: 5-7 天          预计: 3-5 天
```

**关键调整**：
1. 在 Phase 2 **正式增加 "Pipeline 抽象重构"**——把 main.py 拆分为 `main.py`（CLI入口）+ `pipeline.py`（统一处理流程）+ `batch.py`（批量处理），否则后续所有步骤都在给一个 900 行的文件打补丁
2. 在 Phase 2 **正式增加 "核心测试建设"**——至少为 RunTracker、Pipeline 的 happy path 和 error path 写 mock 测试
3. **Step 4 拆成 4a + 4b**，分两次交付，降低单次变更风险
